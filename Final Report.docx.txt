Employee Management System (EMS)
Final Project Report
Course: CS3009 - Software Engineering
Team: Muhammad Faheem, Irtaza Kazmi, Humayun Bilal

1. Introduction
The Employee Management System (EMS) is a Java desktop application built with JavaFX and SQLite to automate core HR operations. The project was completed across three sprints. Sprint 1 established the foundation with authentication, role-based access control, department management, employee lifecycle handling, and deactivation requests. Sprint 2 added leave management, attendance tracking, and employee search. Sprint 3 completed the system with payroll generation, document management, dashboards, and department deletion. Together, these deliverables form a complete role-based HR management platform.

2. Project Vision
The project vision is to provide a secure, integrated, and maintainable HR management platform that supports three distinct roles: Administrator, Manager, and Employee. The system is intended to reduce manual HR work, improve data consistency, and provide controlled access to workforce operations, approvals, and records.

3. Intended Use of the System
3.1 Who will use the system
- Administrator: manages top-level organizational structure and high-authority actions.
- Manager: manages department-scoped employee operations and approvals.
- Employee: uses self-service HR features tied to their own account.
- Secondary stakeholders: HR, Finance, IT Support, and Organizational Management.

3.2 How the system will be used
- Users log in with a username and password.
- The system loads a role-specific home screen after authentication.
- Administrators manage departments, managers, payroll, and approval actions.
- Managers handle department-level employee operations, leave approvals, attendance monitoring, and document moderation.
- Employees use the system for attendance, leave, document uploads, payroll viewing, and dashboard self-service.

3.3 Stakeholder needs
Administrator needs:
- Create departments and assign managers.
- Create manager accounts and oversee organizational structure.
- Approve or reject deactivation requests.
- Directly deactivate employees when needed.
- Generate payroll and manage department records.

Manager needs:
- Add employees within their department.
- Submit deactivation requests for their department employees.
- Approve or reject leave requests.
- View team attendance and search scoped employee data.
- Review and delete employee documents within their department.

Employee needs:
- Log in securely and access only their own data.
- Mark attendance and request leave.
- Upload and open their own documents.
- View personal payroll history and dashboard information.

4. Features / Overall Functionality
4.1 What the system does
- Authenticates users and enforces role-based access control.

6. Structured Specifications
6.1 User Login
Function Name: Login
Module: Authentication
Description: Verifies credentials and loads a role-scoped session.
Inputs: Username, Password
Processing: Query the users table, retrieve the role and employee_id, store the authenticated user in AppSession, and load the correct home screen.
Outputs: Authenticated session and role-based home screen.
Preconditions: A valid user account must exist.
Postconditions: Current user data is stored in AppSession.
Exceptions: Invalid credentials, account not found.

6.2 Add Manager Account
Function Name: Add Employee
Module: Employee Management
Description: Creates a manager employee record and linked login account.
Inputs: Employee ID, Full Name, Job Title, Department, Email, Phone, Salary, Login Username, Password
Processing: Validate required fields, check duplicate employee ID and username, resolve manager details from the selected department, insert the employee record, and insert the user account.
Outputs: Confirmation message and persisted manager record.
Preconditions: Administrator must be logged in.
Postconditions: Manager employee and user records are inserted.
Exceptions: Duplicate ID, duplicate username, invalid email, missing fields.

6.3 Add Employee Account
Function Name: Add Employee
Module: Employee Management
Description: Creates a new employee record and linked login account for a manager's department.
Inputs: Employee ID, Full Name, Job Title, Department, Email, Phone, Salary, Login Username, Password
Processing: Validate required fields, check duplicate employee ID and username, use the manager's assigned department, insert the employee record, and insert the user account.
Outputs: Confirmation message and persisted employee record.
Preconditions: Manager must be logged in and assigned to a department.
Postconditions: Employee and user records are inserted.
Exceptions: Duplicate ID, duplicate username, missing fields, invalid input.

6.4 Create Department
Function Name: Create Department
Module: Department Management
Description: Admin creates a new department and assigns a manager.
Inputs: Department Name, Manager Username
Processing: Validate the department name, validate the manager selection, and insert the department record.
Outputs: Department created and available in employee creation forms.
Preconditions: Administrator must be logged in.
Postconditions: Department record is stored with manager assignment.
Exceptions: Duplicate department name, no manager selected.

6.5 Submit Deactivation Request
Function Name: Submit / Approve / Reject Deactivation Request
Module: Employee Lifecycle
Description: Allows a manager to submit a deactivation request for an employee in their department.
Inputs: Employee ID, Reason
Processing: Validate department ownership, check for an existing pending request, and insert the request with status Pending.
Outputs: Deactivation request stored as pending.
Preconditions: Manager must be logged in and the employee must be active.
Postconditions: Deactivation request is inserted into the database.
Exceptions: Employee not in department, duplicate pending request, employee already inactive.

6.6 Approve or Reject Deactivation Request
Function Name: Submit / Approve / Reject Deactivation Request
Module: Employee Lifecycle
Description: Allows an administrator to approve or reject a pending deactivation request.
Inputs: Request ID, Admin username
Processing: Fetch the linked employee, deactivate the employee if approved, and update the request status to Approved or Rejected.
Outputs: Employee deactivated or request cancelled.
Preconditions: Administrator must be logged in and a pending request must exist.
Postconditions: Request status is updated and the employee record is resolved.
Exceptions: Request not found, request already processed, database error.

6.7 Submit Leave Request
Function Name: Submit Leave Request
Module: Leave Management
Description: Employee submits a leave application for manager review.
Inputs: Leave Type, Start Date, End Date, Reason, Employee ID
Processing: Validate dates, calculate requested days, check leave balance, resolve the manager, and insert the request as Pending.
Outputs: Confirmation message and request routed to the manager.
Preconditions: Employee must be linked to an active employee profile.
Postconditions: Leave request persisted with status Pending.
Exceptions: Insufficient leave balance, invalid dates, overlapping request.

6.8 Approve or Reject Leave Request
Function Name: Approve or Reject Leave Request
Module: Leave Management
Description: Manager approves or rejects a pending leave application.
Inputs: Request ID, Action, Optional comment
Processing: Verify the manager owns the request, deduct leave balance on approval, update the request status, and record decision details.
Outputs: Status updated and employee balance adjusted when approved.
Preconditions: Request exists and is pending.
Postconditions: Leave request status is updated.
Exceptions: Request not found, unauthorized manager, already decided.

6.9 Mark Attendance
Function Name: Mark Attendance
Module: Attendance Tracking
Description: Employee records check-in or check-out for the current day.
Inputs: Employee ID, Current timestamp
Processing: Insert a check-in if no record exists, or update the record with check-out and total hours if a check-in already exists.
Outputs: Check-in or check-out confirmation.
Preconditions: Employee must have an active linked profile.
Postconditions: Attendance record updated in the database.
Exceptions: Already completed today, system time error, employee not found.

6.10 Employee Search
Function Name: Search Employees
Module: Employee Search
Description: Role-scoped employee lookup by keyword.
Inputs: Search text
Processing: Admin queries managers only, while managers query employees within their own department scope.
Outputs: Table of matching employees with ID, name, title, department, and status.
Preconditions: Admin or Manager logged in.
Exceptions: No results found, database error.

6.11 Generate Payroll
Function Name: Generate Payroll
Module: Payroll Management
Description: Administrator calculates and records salary for an employee for a given period.
Inputs: Employee ID, Month, Year, Base Salary, Overtime Hours, Overtime Rate, Deductions, Notes
Processing: Validate the admin role, verify employee status, check month/year range, prevent duplicate payroll, compute gross and net salary, insert the payroll record, and update last_net_salary.
Outputs: Salary slip displayed and payroll stored.
Preconditions: Admin logged in and employee active.
Postconditions: Payroll record persisted and last_net_salary updated.
Exceptions: Non-admin access, duplicate period, inactive employee, negative values.

6.12 Upload Document
Function Name: Upload Document
Module: Document Management
Description: Employee uploads a file to their profile for manager review.
Inputs: Source file
Processing: Validate employee role, confirm file exists, validate extension, copy file into the employee document folder, and insert a record with status Pending.
Outputs: Upload confirmation and document entry in the table.
Preconditions: Employee logged in with linked profile.
Postconditions: File stored on disk and record inserted.
Exceptions: Unsupported file type, file not found, upload failure.

6.13 Approve Document
Function Name: Approve Document
Module: Document Management
Description: Manager marks a document as approved.
Inputs: Document ID, Manager session
Processing: Verify manager role and update the document status to Approved.
Outputs: Document status updated.
Preconditions: Manager logged in and the document belongs to the department.
Postconditions: Document status is updated in the database.
Exceptions: Unauthorized role, document not found.

6.14 Delete Document
Function Name: Delete Document
Module: Document Management
Description: Manager removes a document record and its associated physical file.
Inputs: Document ID
Processing: Verify manager role, fetch file path, delete the DB record, and delete the physical file.
Outputs: Document removed from the table and file deleted from disk.
Preconditions: Manager logged in and the document exists.
Postconditions: Document record removed.
Exceptions: Unauthorized role, document not found, file already deleted.

6.15 Open Document
Function Name: Open Document
Module: Document Management
Description: Employee opens a selected document using the OS default application.
Inputs: Document ID
Processing: Retrieve the file path, verify the file exists, and open it using the desktop handler.
Outputs: File opened in OS default viewer.
Preconditions: Employee logged in and the file exists on disk.
Exceptions: File not found on disk, OS cannot open file type.

6.16 View Admin Dashboard
Function Name: Admin Dashboard
Module: Reporting & Analytics
Description: Provides administrator with global employee statistics and department breakdown.
Inputs: None
Processing: Count all employees, count active employees, and group employees by department with manager names.
Outputs: KPI cards and department table.
Preconditions: Admin logged in.
Exceptions: Database error.

6.17 View Manager Dashboard
Function Name: Manager Dashboard
Module: Reporting & Analytics
Description: Shows manager department-scoped KPIs and pending leave approvals.
Inputs: Manager username
Processing: Count active reports, pending leave requests, employees not yet paid this month, and today's attendance for the team.
Outputs: Department KPI cards and pending leave table.
Preconditions: Manager logged in.
Exceptions: No data.

6.18 View Employee Dashboard
Function Name: Employee Dashboard
Module: Reporting & Analytics
Description: Shows employee personal profile and financial summary.
Inputs: Employee ID
Processing: Load employee details, today's attendance record, leave balance, and current salary summary.
Outputs: Profile grid and pay strip.
Preconditions: Employee logged in with linked profile.
Exceptions: Employee record not found.

6.19 View Own Payroll
Function Name: View Own Payroll
Module: Payroll Management
Description: Allows an employee to view only their own payroll history and salary slips.
Inputs: Employee ID
Processing: Query payroll records for the current employee only, sort them by year and month, and display them in a table.
Outputs: Payroll history view and salary slip details.
Preconditions: Employee must be logged in with a linked profile.
Postconditions: Employee payroll records displayed on screen.
Exceptions: Unauthorized access, no payroll records found, missing employee profile, database error.

6.20 Delete Department
Function Name: Delete Department
Module: Department Management
Description: Allows an administrator to delete an existing department when it no longer has active employees.
Inputs: Department ID, Selected Department
Processing: Verify administrator authorization, check for active employees, block deletion if active employees exist, delete the department, refresh lists, and reset the previous manager to an unassigned state.
Outputs: Department removed confirmation message.
Preconditions: Administrator must be logged in.
Postconditions: Department record removed and manager assignment cleared.
Exceptions: Unauthorized access, department not found, active employees still assigned, database error.

8. NFR Specification
Security: Role predicates should enforce access boundaries and prevent cross-role data leakage.
Usability: Role-aware navigation should show only the actions relevant to the signed-in user.
Performance: Standard database operations should complete within a few seconds on normal hardware.
Reliability: Multi-step database updates should be protected with transactions where needed.
Maintainability: Shared navigation and access checks should be centralized to reduce duplicated logic.
Portability: The application should run on Windows using Java 17, JavaFX 25, and SQLite without external server dependencies.
